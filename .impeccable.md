# Shipwright Design Guidance

This file is the Codex-facing bridge for Shipwright's Impeccable design context. The canonical Impeccable-style project files are `PRODUCT.md` and `DESIGN.md` in this repo root. Read those before UI work.

## Design Context

### Users
Shipwright is for solo builders with dormant, half-finished, or neglected GitHub repositories. They are usually technical enough to understand code and PRs, but they may not know deployment infrastructure, CI, environment variables, lockfiles, hosting requirements, or launch-readiness details deeply. Their job is to turn an existing repo into something deployable, reviewable, and confidence-inspiring without spending days relearning the project.

### Brand Personality
Serious, precise, and useful. Shipwright should feel like a senior deployment engineer reviewing a repo beside the user: calm, specific, and action-oriented. It should avoid hype, gimmicks, vague encouragement, and decorative personality that gets in the way of diagnosis.

### Aesthetic Direction
Use a serious developer-tool aesthetic: operational, dense enough for repeated work, and polished without feeling like a generic SaaS landing page. The product surface should prioritize diagnosis, issues, generated artifacts, and next actions. Avoid neon-dark AI-tool styling, crypto-like visuals, playful mascots, oversized marketing cards, generic gradient hero sections, and decorative UI that does not help the user ship.

### Design Principles
- Lead with deployment truth: show readiness, blockers, warnings, and exact fixes before marketing copy or decoration.
- Make every recommendation inspectable: generated files, deployment plans, and PR bodies should be readable public artifacts.
- Prefer confident utility over magic: explain what Shipwright found, what it changed, and what the user must still verify.
- Keep workflows compact and repeatable: solo builders may run this across many repos, so scanning, comparison, and review should be efficient.
- Treat uncertainty honestly: when Shipwright infers something, label it as inferred and give the user a clear way to validate it.
- Optimize for trust: stable layouts, sober colors, clear severity language, and precise copy matter more than visual novelty.
