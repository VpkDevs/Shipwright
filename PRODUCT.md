# Shipwright Product Context

## Register

Product. Shipwright is an application/workbench where design should serve the user's work. It should feel operational, precise, and trustworthy, not like a brand campaign or decorative landing page.

## Users

Shipwright is for solo builders with dormant, half-finished, or neglected GitHub repositories. They are often capable developers, but they may not know deployment infrastructure, CI, environment variables, lockfiles, hosting requirements, or launch-readiness details deeply. They need help turning an existing repo into something deployable, reviewable, and confidence-inspiring.

## Core Job

The user brings Shipwright a GitHub repository and expects a clear answer:

- What is this project?
- What blocks deployment?
- Which fixes are mandatory, optional, or inferred?
- What files should be generated?
- What should the resulting pull request say and contain?
- What must I verify before shipping?

## Brand Personality

Serious, precise, useful.

Shipwright should feel like a senior deployment engineer reviewing a repo beside the user: calm, specific, and action-oriented. It should avoid hype, gimmicks, vague encouragement, and decorative personality that gets in the way of diagnosis.

## Product Promise

GitHub repo in, deployment diagnosis and reviewable deployment PR out.

Shipwright should not imply that it magically deploys everything correctly. It should communicate confidence through evidence: detected facts, classified issues, generated files, and clear review steps.

## Anti-References

Avoid:

- Neon-dark AI tool styling.
- Purple-to-blue gradients as a default visual identity.
- Crypto-like glow, blur, and glass effects.
- Playful mascots or emoji-driven tool surfaces.
- Oversized marketing-card layouts.
- Generic SaaS hero sections that repeat metrics, feature cards, and vague benefits.
- Monospace typography used merely to signal "developer."

## Design Principles

- Lead with deployment truth: readiness, blockers, warnings, and exact fixes come before decoration.
- Make every recommendation inspectable: generated files, deployment plans, and PR bodies should be readable public artifacts.
- Prefer confident utility over magic: explain what Shipwright found, what it changed, and what the user must still verify.
- Keep workflows compact and repeatable: solo builders may run this across many repos, so scanning, comparison, and review should be efficient.
- Treat uncertainty honestly: when Shipwright infers something, label it as inferred and give the user a clear way to validate it.
- Optimize for trust: stable layouts, sober colors, clear severity language, and precise copy matter more than visual novelty.
